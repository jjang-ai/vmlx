Here's a thinking process that leads to the suggested checklist for opening a PR:

1.  **Understand the User's Need:**
    *   **Context:** The user is about to open a Pull Request (PR) for a *big feature*.
    *   **Goal:** They want to know what to check *before* submitting it to ensure a smooth review process, minimize rework, and increase the chances of acceptance.
    *   **Implicit Need:** "Big feature" implies complexity, potential side effects, and a higher likelihood of review friction. The advice needs to be comprehensive but actionable.

2.  **Categorize the Checklist:**
    To make the information digestible, I should group the checks into logical categories. Good categories would be:
    *   Code Quality & Functionality
    *   Testing
    *   Documentation
    *   Process & Communication
    *   Security & Performance
    *   Final Polish

3.  **Brainstorming Specific Items within Categories:**

    *   *Code Quality:* Does it compile? Are there linting errors? Is the code readable? Did I remove debug logs? Are there TODOs? Is the naming consistent?
    *   *Testing:* Unit tests? Integration tests? E2E? Did I run the test suite? Did I cover edge cases? Did I break existing tests?
    *   *Documentation:* README updates? API docs? Changelog? Comments in code?
    *   *Process:* Is the branch up to date with main? Is the PR description clear? Are there screenshots? Are there related issues linked?
    *   *Security/Performance:* Secrets in code? N+1 queries? Memory leaks? Large file uploads?
    *   *Communication:* Who should review? Is it ready for review (not WIP)?

4.  **Refining for a "Big Feature":**
    *   Since it's a *big* feature, I need to emphasize things that often get overlooked in large changes:
        *   **Scope Creep:** Does this do *only* what it's supposed to do?
        *   **Backwards Compatibility:** Will this break existing APIs or behaviors?
        *   **Migration:** Are there database migrations? Are they reversible?
        *   **Configuration:** Does it require new env vars?
        *   **Rollback Plan:** What if this breaks prod?

5.  **Drafting the Response (Iterative Process):**

    *   *Introduction:* Acknowledge the effort (big feature = hard work) and set the tone (be proactive).
    *   *Section 1: The Code Itself.* Focus on cleanliness, linting, and removing debug stuff.
    *   *Section 2: Testing.* Crucial for big features. Mention coverage and regression.
    *   *Section 3: The PR Description.* This is the most common point of failure. Needs context, how to test, and screenshots.
    *   *Section 4: Impact & Side Effects.* Database changes, config, performance, security.
    *   *Section 5: Process.* Branch hygiene, reviewers, WIP status.
    *   *Section 6: The "Soft" Skills.* Communication, empathy for reviewers.

6.  **Adding Specific Tips/Best Practices:**
    *   *Tip:* Split the PR if it's too big (atomic commits).
    *   *Tip:* Use a checklist in the PR description.
    *   *Tip:* Self-review first.
