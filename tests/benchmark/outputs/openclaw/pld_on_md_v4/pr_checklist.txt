Here's a thinking process that leads to the suggested checklist for opening a Pull Request (PR) for a big feature:

1.  **Understand the Goal:** The user is about to open a Pull Request (PR) for a *big feature*. This implies complexity, potential for breaking changes, significant code changes, and a need for careful review. The goal is to ensure the PR is ready for review, minimizes friction, and increases the chances of getting merged smoothly.

2.  **Identify the Audience:** The advice needs to be actionable, comprehensive, and structured. It should cover technical, communicative, and process-oriented aspects.

3.  **Brainstorm Categories:** What are the key areas to check before hitting "Create Pull Request"?
    *   Code Quality (Linting, formatting, tests)
    *   Functionality (Does it work? Edge cases?)
    *   Documentation (Is it explained?)
    *   Communication (Is the context clear?)
    *   Process/Workflow (Branch strategy, CI/CD)
    *   Security/Performance (Big features often introduce new risks)
    *   Reviewer Preparation (Make the reviewer's job easy)

4.  **Flesh out each category (Drafting content):**

    *   *Code:* Lint? Format? Static analysis? Unused imports?
    *   *Tests:* Unit tests? Integration tests? E2E? Do they pass locally and on CI? Edge cases covered?
    *   *Functionality:* Manual testing? Backwards compatibility? Migration scripts?
    *   *Docs:* README? API docs? Changelog? User-facing changes?
    *   *PR Description:* Title? Summary? Screenshots? Links to tickets? "Why" vs "How"?
    *   *Cleanup:* Rebase? Squash? Remove debug logs? Secrets?
    *   *Big Feature Specifics:* Breaking changes? Migration strategy? Rollback plan? Performance impact?

5.  **Structure and Refine:** Organize the brainstormed points into a logical flow. A checklist format works best. Group them into "Before You Commit," "Before You Push," and "Before You Submit."
