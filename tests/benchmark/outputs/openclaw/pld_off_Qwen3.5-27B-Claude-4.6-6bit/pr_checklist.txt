This is a comprehensive checklist to ensure your PR is high-quality, easy to review, and has the best chance of being merged smoothly.

### 1. Code Quality & Functionality
*   **Does it pass all tests?** Run your local test suite. Ensure 100% pass rate.
*   **Did you add tests for the new feature?** If you added logic, you need unit/integration tests to cover it.
*   **Did you run the linter/formatter?** Ensure your code matches the project's style (e.g., Prettier, ESLint, Black, GoFmt).
*   **Did you remove debug code?** Search for `console.log`, `debugger`, `print`, or temporary variables you forgot to delete.
*   **Does it actually work?** Manually test the feature in a staging environment or locally. Don't rely solely on automated tests.

### 2. "Big Feature" Specifics
*   **Is it atomic?** If the PR is massive (e.g., 2000+ lines), consider splitting it. A PR should ideally do one thing. If it mixes database migrations, UI changes, and API logic, can it be broken down?
*   **Are there breaking changes?** Did you change an API signature or a database schema that will break other parts of the app or dependent services?
*   **Are there database migrations?** If yes, are they reversible? Have you tested running them on a local database with production-like data?
*   **Configuration/Environment Variables:** Did you add new env vars? Are they documented? Do they have sensible defaults?

### 3. Documentation
*   **Is the PR description clear?** A good PR description is crucial.
    *   **Summary:** What does this do?
    *   **Context:** Why are we doing this? (Link to Jira/Ticket/Issue)
    *   **How to test:** Give the reviewer step-by-step instructions on how to verify the feature.
    *   **Screenshots/Recordings:** If it's UI, show it!
*   **Is the code documented?** Did you add comments for complex logic? Did you update JSDoc/TypeDoc?
*   **Did you update the README?** If the feature changes how the project is installed or used, the README must reflect that.

### 4. Housekeeping
*   **Did you rebase/merge main?** Pull the latest changes from the `main` (or `develop`) branch and resolve any conflicts *before* opening the PR. This prevents merge hell later.
*   **Did you squash commits?** If you have 50 "WIP" commits, squash them into logical, atomic commits with clear messages.
*   **Are dependencies updated?** Did you commit the `package-lock.json` or `requirements.txt`?

### 5. The "Pre-Flight" Review
*   **The "5-Minute" Rule:** Wait 5 minutes after writing your code, then read it again. You will catch obvious typos and logic errors.
