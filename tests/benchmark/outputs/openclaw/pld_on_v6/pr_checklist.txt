Opening a Pull Request (PR) for a large feature is a significant moment. A "big" feature introduces more risk and complexity than a small bug fix.

Here is a comprehensive checklist to ensure your PR is ready for review, categorized by the type of check.

### 1. The "Hard" Technical Checks
Before you even think about the UI, ensure the code is solid.

*   **The Build:** Does it compile/build without errors?
*   **The Tests:**
    *   Do **all** existing tests pass? (Ensure you didn't break legacy code).
    *   Do **your** new tests pass?
    *   Is the coverage sufficient? (Don't just cover the "happy path"; cover edge cases and error states).
*   **Linting & Formatting:** Run your linter (ESLint, Pylint, RuboCop, etc.) and formatter (Prettier, Black). A PR full of style errors distracts reviewers from the logic.
*   **Local Verification:** Have you manually tested the feature end-to-end in a local environment? Don't rely solely on unit tests.

### 2. The "Big Feature" Specifics
Since this is a large feature, you need to consider the broader implications.

*   **Database Migrations:** If you changed a database schema, have you included the migration scripts? Have you tested the rollback?
*   **Backwards Compatibility:** Will this break existing APIs, configurations, or user data?
*   **Performance:** Have you checked for N+1 queries, memory leaks, or heavy computations that might slow down the system?
*   **Security:** Are you handling user input safely? Are there new permissions or access controls needed?
*   **Feature Flags:** Is the feature behind a toggle? This allows reviewers to merge the code without exposing the incomplete feature to users immediately.

### 3. The "Reviewer Experience" (Crucial for Big PRs)
Large PRs are often rejected or ignored because they are too daunting to read. Make it easy for your team.

*   **The Description:**
    *   **Context:** Why are we building this? Link to the Jira ticket, Figma design, or spec doc.
    *   **Summary:** What does this code actually *do*?
    *   **How to Test:** Give the reviewer step-by-step instructions on how to verify the feature manually.
*   **Screenshots/Recordings:** If it involves UI, include a GIF or screenshots. "A picture is worth a thousand lines of code."
*   **Atomic Commits:** Have you squashed your "WIP" (Work In Progress) and "fix typo" commits? The history should tell a clean story of the feature's development.
*   **Scope:** Is it *too* big? If the PR is 2,000+ lines, consider if it can be split into smaller, logical PRs (e.g., Database changes first, then API, then UI).
