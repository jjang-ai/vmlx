Opening a Pull Request (PR) for a large feature is a significant step. A "big" feature introduces a higher risk of breaking existing functionality and can be difficult for reviewers to digest.

Here is a comprehensive checklist to ensure your PR is ready for review, categorized by the type of check.

### 1. The "Green Light" Checks (Automation)
Before you even think about clicking the button, ensure your local environment is clean.

*   **Tests Pass:** Run the full test suite locally. For a big feature, this is non-negotiable. Ensure you have tests for the new feature *and* that existing tests still pass (regression testing).
*   **Linting & Formatting:** Run your linter (ESLint, Pylint, etc.) and formatter (Prettier, Black). You don't want the reviewer to nitpick about semicolons when they should be looking at logic.
*   **Build Success:** If you have a build step (compiling TypeScript, bundling assets), make sure it completes without errors or warnings.
*   **Branch is Up to Date:** Rebase or merge your feature branch with the main branch (e.g., `main` or `develop`). Resolve any conflicts locally before pushing.

### 2. The Code Quality Checks
*   **Dead Code Removal:** Did you leave behind `console.log` statements, commented-out blocks of code, or unused imports? Clean them up.
*   **Secrets & Keys:** Double-check that you haven't accidentally committed API keys, passwords, or `.env` files.
*   **Database Migrations:** If you changed a database schema, have you included the migration scripts? Are they idempotent (safe to run multiple times)?
*   **Feature Flags:** If the feature is massive, is it behind a Feature Flag so you can turn it off if it breaks?

### 3. The "Reviewer Experience" Checks
Big PRs are hard to review. Make it easy on your team.

*   **The Description:**
    *   **Context:** Why are we building this? Link to the Jira ticket or Product Requirement Document (PRD).
    *   **Changes:** What files were touched and why?
    *   **How to Test:** Give the reviewer specific steps to verify the feature manually.
    *   **Screenshots/Recordings:** If it's a UI feature, include a GIF or screenshots of the before and after.
*   **Atomic Commits:** Have you squashed your commits? A history of "WIP", "fix", "fix fix", "update" is messy. Squash them into logical commits
