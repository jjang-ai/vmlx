**What it is:** The KV cache stores the **key and value matrices** computed during the attention mechanism for **the current generation sequence**.

**Purpose:** Avoid recomputing attention keys and values for previously generated tokens during autoregressive decoding.

**How it works:**
```
Generation Step 1: Compute K₁, V₁ → Cache them
Generation Step 2: Compute K₂, V₂ → Cache them | Use K₁,V₁ for attention
Generation Step 3: Compute K₃, V₃ → Cache them | Use K₁,V₁,K₂,V₂ for attention
```

**Scope:** Per-request, per-generation. Cleared after each request completes.

---

## Prefix Cache
**What it is:** A prefix cache stores KV states for **common prefixes** that appear across **multiple requests**.

**Purpose:** Share computation across requests that have identical or overlapping prompt prefixes.

**How it works:**
```
Request 1: "Explain quantum physics..." → Cache prefix KV
Request 2: "Explain quantum physics..." → Reuse cached prefix KV!
Request 3: "Explain classical physics..." → Different prefix, no reuse
```

**Scope:** Global/shared across requests. Persists until evicted.

---

## Key Differences

| Aspect | KV Cache | Prefix Cache |
|--------|----------|--------------|
| **Granularity** | Per-request | Shared across requests |
| **Lifetime** | Duration of single generation | Persists across multiple requests |
| **Reuse Scope** | Within one sequence | Across different sequences with same prefix |
| **Memory Location** | Typically GPU memory | Can be CPU/GPU, sometimes disk-backed |
| **Eviction** | Automatic (after generation) | Policy-based (LRU, size limits) |
| **Benefit** | O(n²) → O(n) per sequence | Avoids recomputing common prompts |
