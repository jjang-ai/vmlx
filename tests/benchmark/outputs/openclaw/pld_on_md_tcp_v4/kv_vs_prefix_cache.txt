2.  **Define "KV Cache":**
    *   *What is it?* It's the standard mechanism for caching Key and Value matrices in Transformer models during autoregressive generation.
    *   *Purpose:* Avoid recomputing attention for previous tokens. Speeds up decoding.
    *   *Scope:* Specific to a *single* request/session. It grows as the sequence length grows.
    *   *Lifecycle:* Created per request, discarded when request finishes.
    *   *Key Characteristic:* It's the *mechanism* of caching.

3.  **Define "Prefix Cache":**
    *   *What is it?* It's a higher-level optimization strategy (often used in systems like vLLM, TGI, or research papers like "Prefix Attention").
    *   *Purpose:* Share computation across *multiple* requests that share common prefixes (e.g., system prompts, repeated user inputs).
    *   *Scope:* Shared across requests.
    *   *Lifecycle:* Stored in a pool, reused by different requests.
    *   *Key Characteristic:* It's the *strategy* of reusing KV caches.

4.  **Identify the Relationship:**
    *   Prefix caching *relies* on KV caching. You can't have prefix caching without the concept of KV caching.
    *   KV caching is the low-level implementation detail (per token).
    *   Prefix caching is the high-level system optimization (per request batch).
