Opening a Pull Request (PR) for a "big feature" is a high-stakes moment. The goal is to minimize friction, ensure quality, and make the reviewer's life easy.

Here is a comprehensive checklist to run through before you hit that "Create Pull Request" button.

### 1. Code Quality & Functionality
*   **Run the Linter/Formatter:** Before you push, run your project's linter (e.g., ESLint, Pylint, RuboCop) and formatter (Prettier, Black). You don't want the PR to be rejected because of trailing commas or line lengths.
*   **The "Green" Build:** Run the full test suite locally. If it takes too long, at least run the tests relevant to your feature. Ensure **100% pass rate**.
*   **Add/Update Tests:** Did you write unit tests for the new logic? Did you write integration tests? If the feature is complex, a lack of tests will be the first thing a reviewer critiques.
*   **Manual Testing:** Did you actually use the feature? Try to break it. Edge cases often hide in manual testing.

### 2. The "Big Feature" Specifics
*   **Rebase/Merge Main:** Since it's a big feature, you likely worked on it for a while. Merge or rebase the `main` (or `develop`) branch into your feature branch. **Resolve conflicts now**, not during the review.
*   **Check for "Dead" Code:** Did you leave behind `console.log`, `debugger`, or commented-out blocks of code you thought you'd need later? Remove them.
*   **Database Migrations:** If you touched the database, are the migration scripts idempotent (safe to run multiple times)? Do they roll back cleanly?

### 3. Documentation
*   **Update READMEs:** If the feature changes how the app is configured or used, update the documentation.
*   **API Docs:** If you exposed new endpoints, are they documented (Swagger/OpenAPI)?
*   **Inline Comments:** For complex logic, did you leave comments explaining *why* you did something, not just *what* you did?

### 4. The PR Description (The Pitch)
A big PR needs a big description. If you just write "Added feature," it will be ignored or reviewed poorly.
*   **The "Why":** Explain the context. What problem does this solve?
*   **The "How":** Briefly explain the technical approach.
*   **Proof:** If you change the UI, take **screenshots or record a Loom video** showing the feature in action.
*   **Testing Instructions:** Tell the reviewer exactly how to test this. (e.g., "1. Go to settings, 2. Click X, 3. Observe Y").
